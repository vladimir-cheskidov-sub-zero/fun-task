# Categories

`Categories` — CLI-приложение на PHP 7.4 для обработки дерева категорий витрины. Проект читает `data/categories.json`, строит доменную модель и выполняет три сценария через Visitor:

- построение публичного меню;
- аудит проблем видимости и конфигурации;
- экспорт документов для поискового индекса.

Этот README одновременно описывает текущее состояние приложения и фиксирует результаты практики вайбкодинга: что получилось хорошо, где возникали ошибки, какие решения пришлось пересматривать и какие ограничения управляли разработкой.

## Краткий итог практики

- Получился рабочий CLI-инструмент с тремя отдельными командами, ручной DI-сборкой, доменной моделью с инвариантами и заметным покрытием unit-тестами.
- Самый сильный эффект дали точные prompt-ы, которые ссылались на конкретные файлы, ограничения и инварианты; самые слабые — общие ревью-промпты и повторные микро-рефакторинги одной и той же темы.
- Главные источники повторной работы: неоднозначности в `ai/specification.md`, постепенное ужесточение правил из `agents.md` и отсутствие заранее зафиксированных golden/snapshot-контрактов для CLI.
- Репозиторий особенно ценен тем, что сохраняет не только код, но и историю разработки в `ai/prompts/`: это позволяет оценивать не результат “в вакууме”, а реальный ход агентной разработки.

## Что умеет приложение

- `categories:menu` строит публичное меню с учетом тегов `menu`, `hidden`, `restricted:*`, `region:*`.
- `categories:audit` собирает проблемные категории и причины попадания в аудит.
- `categories:search-export` формирует JSON-экспорт для поискового индекса в `export/<timestamp>-index.json`.
- Все команды запускаются через `php app ...`.
- Входные данные берутся из JSON-дерева категорий, которое поддерживается как внешний входной артефакт в `data/`.

## Архитектура приложения

Проект следует четырехслойной схеме, зафиксированной в `agents.md` и `ai/specification.md`.

### Domain

Слой `Domain` отвечает за бизнес-смысл дерева категорий:

- `Category` — агрегат, который хранит `id`, `name`, `tags`, `children` и гарантирует инварианты дерева.
- `Tag` — value object, который парсит строковый тег, определяет его тип и параметр.
- `TagType`, `Region`, `RestrictedVisibility` — перечисления на базе MyCLabs Enum.
- `CategoryVisitor` — контракт Visitor-а с методами `enter()` и `leave()`.
- `MenuBuilderVisitor`, `VisibilityAuditVisitor`, `SearchIndexExportVisitor` — доменные сценарии обхода.
- `MenuCategoryVisibilitySpecification` — отдельная спецификация видимости для меню.

Ключевая идея: домен знает факты о категории, а visitor-ы знают, как эти факты использовать в конкретном сценарии.

### Application

Слой `Application` оркестрирует сценарии:

- `BuildMenuService`
- `AuditVisibilityService`
- `SearchIndexExportService`

Каждый use case:

- принимает query-объект;
- создает нужный visitor;
- загружает дерево через контракт `CategoryHydrator`;
- вызывает обход дерева;
- мапит доменный результат в DTO через assembler;
- переводит доменные исключения в исключения application-слоя.

Это защищает Bridge от знания доменных деталей и делает CLI-команды тонкими.

### Infrastructure

Слой `Infrastructure` реализует внешние интеграции:

- `JsonFileCategoryHydrator` читает JSON-файл;
- валидирует структуру;
- строит дерево `Category`;
- различает ошибки чтения файла, ошибки структуры и доменные нарушения.

Важно, что инфраструктурный код зависит от доменного интерфейса `CategoryHydrator`, а не наоборот.

### Bridge

Слой `Bridge` содержит код, привязанный к Symfony Console:

- `CategoriesMenuCommand`
- `CategoriesAuditCommand`
- `CategoriesSearchExportCommand`
- `ApplicationFactory`
- `ContainerFactory`
- `EntryPointRunner`
- `ConsoleOutputEscaper`

Bridge знает, как получить аргументы CLI, как вывести DTO пользователю и как завершить приложение с единообразной обработкой ошибок.

### Сквозной поток выполнения

Для любой команды поток одинаковый:

1. `php app ...` запускает `EntryPointRunner`.
2. `EntryPointRunner` поднимает DI-контейнер и Symfony `Application`.
3. Конкретная команда Bridge валидирует входные аргументы/опции и создает query.
4. Application service гидрирует дерево, запускает visitor и собирает DTO.
5. Bridge рендерит DTO в текст или пишет файл.
6. Любое неотловленное исключение поднимается в `EntryPointRunner`, где превращается в безопасное CLI-сообщение.

## Технический дизайн и ключевые решения

### 1. Иммутабельная доменная модель

`Category`, `CategoryTags` и `ChildCategories` устроены так, чтобы не оставлять объект в невалидном состоянии. Операции вроде `rename()`, `addTag()` или `addChild()` возвращают новый объект, а не мутируют старый.

Плюсы:

- проще держать инварианты;
- проще тестировать;
- легче reason about в visitor-ах.

Цена:

- больше вспомогательных объектов;
- выше “цена” каждого изменения доменной модели;
- для большой модели может стать заметнее накладной расход на создание объектов.

### 2. Семантические предикаты в `Category`

По ходу развития проекта знания о тегах постепенно переносились из visitor-ов в `Category`:

- `isRoot()`
- `isMenuItem()`
- `isHidden()`
- `isPromoted()`
- `isSearchable()`
- `hasRestrictedVisibility()`
- `regions()`
- `isVisibleForRegion()`

Это важное архитектурное решение. Изначальная спецификация местами подталкивала держать всю логику обработки тегов в visitor-ах, но практика показала, что постоянное сканирование тегов снаружи создает дублирование и ломает инкапсуляцию. В итоге агрегат стал источником истины о самом себе, а visitor-ы — потребителями этих знаний.

### 3. Visitor с парой `enter/leave`, а не с одним `visit`

Такой контракт оказался удачным:

- `MenuBuilderVisitor` хранит стек видимости ветки;
- `VisibilityAuditVisitor` хранит стек пути для иерархии;
- `SearchIndexExportVisitor` просто накапливает документы в порядке обхода.

То есть пара `enter/leave` дала простой способ держать контекст обхода без отдельного walker-а и без засорения `Category` сценарной логикой.

### 4. Спецификация для правил меню

Правило “должна ли категория попасть в меню” вынесено в `MenuCategoryVisibilitySpecification`.

Это хорошее решение по нескольким причинам:

- visitor перестал быть перегруженным;
- правила стали переиспользуемыми и тестируемыми;
- бизнес-условия меню отделены от механики обхода дерева.

### 5. DTO + Assembler вместо утечки домена в CLI

Bridge получает не доменные объекты, а DTO:

- `MenuDto`
- `VisibilityAuditDto`
- `SearchIndexExportDto`

Сборка DTO вынесена в assembler-ы. Это делает формат вывода независимым от внутренней формы доменных классов и позволяет менять CLI, не ломая домен.

### 6. Явное маппирование исключений по слоям

Application layer не пробрасывает доменные исключения наружу как есть. Вместо этого use case-ы создают:

- `BuildMenuFailed`
- `AuditVisibilityFailed`
- `SearchIndexExportFailed`

При этом `previous` сохраняется, что важно для `--verbose` режима и отладки. Это одно из самых полезных решений в проекте: оно одновременно делает CLI-сообщения понятнее и не теряет stack trace.

### 7. Единая обработка ошибок в entry point

`EntryPointRunner` перехватывает любые `Throwable`, печатает sanitized message и возвращает ненулевой код завершения. При `--verbose` выводится полная информация об исключении.

Практический эффект:

- приложение не падает “сырой” ошибкой;
- все команды ведут себя единообразно;
- правило из `agents.md` про обработку исключений соблюдается централизованно.

### 8. Ручная DI-конфигурация

`ContainerFactory` собирает контейнер вручную, через явные `register()` и `Reference`.

Плюсы:

- полностью прозрачно, какие зависимости есть;
- удобно для маленького CLI-приложения;
- хорошо сочетается с требованием явной архитектуры.

Минусы:

- конфигурация многословна;
- по мере роста проекта число ручных регистраций будет расти линейно;
- нет автоматической защиты от забытой регистрации, кроме тестов и запуска приложения.

### 9. Вывод поискового экспорта как инфраструктурный/bridge-артефакт

Use case `SearchIndexExportService` возвращает DTO, а запись файла делает `CategoriesSearchExportCommand`.

Это удачное разделение:

- домен и application не знают о файловой системе;
- формат JSON и путь `export/<timestamp>-index.json` остаются на уровне CLI/integration concern;
- write-side легко заменить при необходимости.

## Что показала практика вайбкодинга

### Успехи

- Функциональная цель достигнута: три сценария из спецификации реализованы.
- Архитектурный рисунок проекта читается: слои, границы, DTO, visitor-ы и DI не смешаны.
- История prompt-ов делает разработку аудируемой.
- В проекте много unit-тестов, причем они лежат рядом с архитектурными решениями, а не только с happy path.
- Рефакторинги шли не хаотично, а по понятным осям: инкапсуляция, exception contracts, layer isolation, CLI contract hardening.

### Ошибки и источники повторной работы

- Несколько областей пришлось чинить серией мелких prompt-ов, потому что важные решения не были уточнены заранее:
  - опция `--region`;
  - стратегия обработки enum-ов;
  - санитизация консольного вывода;
  - распределение знаний о тегах между `Category` и visitor-ами.
- Некоторые prompt-ы были слишком широкими или слишком короткими, поэтому давали скорее ревью-эффект, чем точный change request.
- README и высокоуровневая документация появились слишком поздно — это типичный побочный эффект вайбкодинга, когда энергия уходит в код и локальные правки.

### Недоработки и упущения

- Нет отдельного ADR/decision log: архитектурные решения разбросаны по prompt log-ам.
- Нет golden/snapshot-тестов на CLI-вывод и поисковый экспорт; есть unit-тесты, но меньше сквозных контрактов.
- `agents.md` и `ai/specification.md` иногда задают конфликтующие акценты, а машинно-проверяемого слоя правил между ними нет.
- Ручная DI-конфигурация масштабируется ограниченно.
- В репозитории до этого README не было документа, который бы собирал в одно место архитектуру, установку и ретроспективу разработки.

## `agents.md` как управляющий документ разработки

Ссылка: [agents.md](agents.md)

### Что делает `agents.md`

`agents.md` — это не просто coding style guide. В этом проекте он выступает сразу в четырех ролях:

- технические ограничения среды;
- архитектурный guardrail;
- процессный чеклист для агента;
- журналирующее требование для prompt-ов.

То есть файл управляет не только “как писать код”, но и “как вообще должна идти разработка”.

### Как `agents.md` повлиял на ход разработки

Положительное влияние:

- зафиксировал PHP 7.4 и не дал уплыть в новые языковые конструкции;
- удержал проект в четырехслойной архитектуре;
- заставил явно описывать `@throws`, что заметно подняло качество error handling;
- требование про `previous` привело к серии точечных исправлений и улучшило диагностику;
- требование логировать prompt-ы создало отличный материал для ретроспективы;
- требование покрывать DI-сервисы и доменные модели тестами дало крепкую unit-test базу;
- требование не ронять приложение неотловленными исключениями привело к полезной централизации в `EntryPointRunner`.

Негативное или замедляющее влияние:

- файл смешивает архитектурные, процессные и эксплуатационные правила, из-за чего возрастает когнитивная нагрузка на агента;
- часть правил требует интерпретации, а не буквально проверяется;
- некоторые требования стимулировали не feature work, а серию пост-фактум аудитов и доработок;
- запрет на nullable/default-параметры в сочетании с опциональной CLI-семантикой стал источником цепочки корректировок вокруг `--region`.

### Насколько `agents.md` был полезен

Как инструмент дисциплины — очень полезен. Как единственный источник правил — перегружен.

Практический вывод:

- для агентной разработки такой документ нужен;
- но лучше делить его минимум на два артефакта:
  - архитектурно-кодовые правила;
  - процессные/операционные требования для агента.

Именно в текущем виде `agents.md` помог удержать качество, но одновременно стал причиной заметной доли повторной работы.

## `ai/specification.md` как спецификация для AI-агента

Ссылка: [ai/specification.md](ai/specification.md)

### Пригодность спецификации

В качестве стартовой спецификации для AI-агента документ хорош. Он содержит:

- цель проекта;
- входную модель данных;
- набор тегов;
- технологические ограничения;
- архитектурную рамку;
- три пользовательских сценария;
- CLI-контракт;
- нефункциональные требования;
- критерии готовности.

Этого достаточно, чтобы агент смог:

- быстро собрать правильный каркас;
- понять, какие объекты нужны в домене;
- реализовать visitor-ы;
- оформить три CLI entry point-а.

### Почему спецификация не полностью достаточна

Несмотря на качество, у документа есть важные неоднозначности:

- В разделе архитектуры сказано, что логика обработки тегов должна жить в visitor-ах и связанных сервисах, но последующая эволюция проекта показала, что часть этой логики правильнее держать в `Category` как в источнике доменных фактов.
- В разделе загрузки дерева сказано, что гидратор возвращает коллекцию корневых узлов, а фактическая модель и тестовые данные построены вокруг одного технического `root`.
- В примере меню показан `Каталог`, но в явных допущениях `root` объявлен техническим контейнером и “может не выводиться”; это оставляет свободу, которая потом потребовала уточнения.
- Спецификация задает допустимые значения `--region`, но не до конца проговаривает поведение при отсутствии опции, пустой строке и служебных sentinel-значениях.
- Документ хорошо задает “что должно быть”, но слабее определяет “где именно должна лежать логика” в спорных местах между агрегатом, visitor-ом и спецификацией.

### Анализ пригодности для агентной разработки

Оценка как стартовой спецификации: высокая.

Оценка как единственной спецификации на весь цикл разработки: средняя.

Хорошо работает:

- постановка фич;
- первичное проектирование;
- acceptance-критерии верхнего уровня.

Хуже работает:

- тонкие контрактные детали CLI;
- распределение ответственности между классами;
- edge cases, которые влияют на clean architecture.

### Как `ai/specification.md` повлиял на ход разработки

Позитивное влияние:

- задала весь feature scope проекта;
- напрямую определила набор visitor-ов и команд;
- подтолкнула к использованию enum-ов и VO;
- помогла держать фокус на порядке обхода, формате вывода и инвариантах дерева.

Негативное или ограничивающее влияние:

- некоторые формулировки оказались слишком общими и позже пересматривались серией refactor/review prompt-ов;
- часть решений пришлось уточнять уже после появления рабочего кода;
- спецификация не породила сразу набор точных сквозных тестов, поэтому несколько дефектов нашли уже в review-режиме.

Главный вывод: `ai/specification.md` отлично подходит как “product + architecture brief”, но для агентной разработки следующего уровня ему нужен второй слой — набор более жестких контрактов и edge cases.

## Итоговый анализ эффективности prompt-подхода

### Что работало лучше всего

- prompt-ы, которые:
  - ссылались на конкретный файл и строку;
  - содержали ожидаемое поведение, а не только “переделай”;
  - требовали regression/unit test;
  - фиксировали архитектурное намерение, а не только симптом.

Именно такие prompt-ы дали самые сильные улучшения: `exception-previous-audit`, `structure-catch-narrowing`, `reject-empty-region-option`, `exclude-root-from-search-export`, `move-region-knowledge-to-category`.

### Что работало хуже

- широкие prompt-ы вроде “пройдись”;
- prompt-ы без четкого acceptance-критерия;
- повторные микро-рефакторинги одной и той же темы без заранее принятого решения.

Это хорошо видно по веткам:

- enum validation;
- console sanitization;
- region CLI contract.

### Оценка экономии времени

Это оценка, а не точный тайм-трек. Она опирается на историю `git` и структуру изменений:

| Метрика | Наблюдение | Что это значит |
| --- | --- | --- |
| Период разработки | `2026-03-29 02:24` -> `2026-04-01 03:31` | От init до финальной самокоррекции прошло около 73 часов календарного времени. |
| Объем изменений | `18` коммитов, `11,999` вставок, `483` удаления | Для CLI на PHP 7.4 это крупный объем, причем вместе с тестами, DI, CLI-слоем и документацией. |
| Крупные вертикальные срезы | `Boilerplate` `+5154`, `Domain model` `+1262`, `Menu` `+1035`, `Audit` `+1249`, `Search export` `+871` | Prompt-подход позволял проталкивать сразу целые функциональные срезы, а не собирать их вручную по файлам. |
| Число prompt log-ов | `41` с учетом этой записи | Экономия была не магической: заметная часть времени ушла на corrective loop и микрорефакторинги. |
| Активное время по commit-clusters | примерно `11.5-18.6` часа при капе интервала между коммитами в `1-2` часа | Это правдоподобный диапазон именно активной работы, если не считать паузы между сессиями. |

Сравнение с ручной реализацией, где ИИ используется только как советчик:

- для такого же объема кода, тестов и архитектурных слоев разумно ожидать около `20-32` часов активной работы;
- основная потеря времени была бы не в придумывании, а в ручном переносе решений в код, синхронизации нескольких файлов и дописывании тестов после каждого изменения;
- prompt coding в этом репозитории, по консервативной оценке, сэкономил около `8-13` часов активного времени;
- в относительных величинах это примерно `30-45%` экономии, или ускорение порядка `1.4-1.8x`.

Практический вывод: в этом проекте prompt coding дал не десятикратный выигрыш, а хороший инженерный буст на уровне одной-двух рабочих смен. Это согласуется с историей `ai/prompts/`: сильнее всего экономили точные change request-ы на большие вертикальные срезы, а заметную часть выигрыша съедали повторные уточнения по enum-ам, sanitization и region contract.

### Главный вывод

Вайбкодинг в этом проекте оказался эффективным там, где у агента были:

- жесткие локальные ограничения;
- понятный артефакт для проверки;
- тест как часть задачи.

Он оказался менее эффективным там, где:

- спецификация оставляла архитектурную двусмысленность;
- правила из `agents.md` были сформулированы широко;
- change request был дан как общий ревью-сигнал, а не как точное требование.

Именно поэтому этот проект хорошо показывает реальную зрелость практики:

- успехи видны в архитектуре и покрытии;
- ошибки видны в цепочках corrective prompt-ов;
- недоработки видны в документации, автоматизации и некоторых повторных рефакторингах.
