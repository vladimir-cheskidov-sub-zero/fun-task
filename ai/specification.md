# Спецификация: дерево категорий витрины

## Цель

Нужно реализовать CLI-приложение на PHP 7.4, которое читает дерево категорий из JSON-файла, строит объектную модель и выполняет над деревом три операции через паттерн Visitor:
- построение публичного меню;
- аудит проблем видимости и конфигурации;
- экспорт документов для поискового индекса.

Решение должно быть чистым, расширяемым и явно демонстрировать корректное применение Visitor.

## Контекст

Есть дерево категорий витрины. Контент-менеджеры поддерживают его в JSON-файле. Каждый узел содержит:
- `id` — уникальный строковый идентификатор;
- `name` — отображаемое название;
- `tags` — набор тегов;
- `children` — дочерние категории.

## Теги

### Простые
- `root`
- `menu`
- `promo`
- `hidden`
- `searchable`

### Параметризованные
- `restricted:staff-only`
- `restricted:18plus`
- `region:kg`
- `region:ru`

## Технологические ограничения

- PHP 7.4;
- CLI-приложение;
- Symfony Console для команд, аргументов и опций;
- Symfony dependency injection component для конфигурирования кажого сервиса;
- MyCLabs Enum для перечислений, где это уместно;
- PSR-4 namespace `FunTask\\` => `src/`;
- статический анализ: PHPStan, PHP CS Fixer и Deptrac;
- нельзя использовать сторонние библиотеки для работы с деревьями и Visitor;
- нельзя модифицировать `vendor/` и `cache/`;
- доменные модели не должны позволять создавать себя в невалидном состоянии
- каждый созданный DI сервис или доменная модель должны быть покрыты unit test-ами.

## Архитектурные требования

### Общие
- Логика обработки тегов должна находиться в Visitor и связанных с ним сервисах.
- `Category` не должен содержать прикладные методы вроде `buildMenu()` или `toSearchDoc()`.
- `Tag` должен быть Value Object.
- Решение не должно быть собрано в одном файле.

### Слои
- `Domain` — `Category`, `Tag`, enum-ы, контракт Visitor.
- `Application` — сценарии запуска Visitor, DTO, orchestration.
- `Infrastructure` — чтение файла, парсинг JSON, гидрация дерева.
- `Bridge` — команды Symfony Console и все другие framework-specific элементы.

## Доменная модель

### Category

`Category` хранит:
- `id`;
- `name`;
- коллекцию `Tag`;
- коллекцию дочерних `Category`.

`Category` отвечает только за:
- хранение данных узла;
- предоставление доступа к данным;
- входную точку обхода `accept(CategoryVisitor $visitor): void`, если обход организован внутри узла.

### Tag

`Tag` как Value Object должен:
- хранить исходное значение тега;
- разбирать тип и параметр;
- сравниваться по значению;
- валидировать допустимые комбинации.

Желательно выделить enum-ы для:
- типа тега;
- значения `restricted`;
- региона.

## Загрузка дерева

Нужен компонент, который:
- принимает путь к JSON-файлу;
- читает и декодирует JSON;
- валидирует структуру;
- рекурсивно строит дерево `Category`;
- возвращает коллекцию корневых узлов.

Ошибка выполнения команды возникает, если:
- файл не существует;
- JSON невалиден;
- отсутствуют обязательные поля;
- поля имеют неверный тип;
- встречены неизвестные теги или параметры;
- найдены дубликаты `id`.

## Visitor-модель

Допустимы два способа обхода:
- обход внутри `Category::accept()`;
- отдельный `TreeWalker`, который вызывает visitor на каждом узле.

Оба варианта допустимы, если:
- обход дерева централизован;
- сценарная логика находится не в `Category`, а в Visitor;
- Visitor умеет накапливать и возвращать результат.

Возможный контракт:
- `visit(Category $category): void`;
- либо `enter(Category $category): void` и `leave(Category $category): void`.

## Сценарий 1: MenuBuilderVisitor

### Назначение

Построить структуру публичного меню.

### CLI entry point

- аргумент `path` — путь к JSON-файлу;
- опция `--adult` — `true` / `false` / `1` / `0`;
- опция `--region` — `kg` / `ru`;
- опция `--staff` — `true` / `false` / `1` / `0`.

Все опции должны валидироваться.

### Правила выборки

Категория не должна попасть в публичное меню, если:
- имеет тег `hidden`;
- имеет тег `restricted:staff-only` и команда запущена без `--staff=true`;
- имеет тег `restricted:18plus` и команда запущена с `adult=false`;
- имеет тег `region:ru` и команда запущена с `region=kg`;
- имеет тег `region:kg` и команда запущена с `region=ru`.

Дополнительно:
- в меню включаются только категории с тегом `menu`;
- категория с тегом `root` считается техническим корнем дерева и может не выводиться как обычный пункт меню;
- если узел исключен из меню, его потомки исключаются из меню так же;
- порядок категорий должен совпадать с порядком во входном JSON.

### Формат вывода

Вывод человекочитаемый, древовидный, в произвольном формате, имитирующем итоговое меню.

Пример:

```text
Каталог
- Электроника
  - Смартфоны
    - iOS-смартфоны
- Бытовая техника
  - Уборка
```

## Сценарий 2: VisibilityAuditVisitor

### Назначение

Собрать проблемные категории.

### Категория попадает в выборку, если

- имеет тег `hidden`;
- имеет тег `promo`, но не имеет тега `searchable`;
- имеет любой тег `restricted:*` и одновременно имеет тег `menu`.

### Проблемы

Для каждой категории нужно вывести одну или несколько причин:
- `hidden` — скрыта и требует пересмотра;
- `promo_without_searchable` — участвует в промо, но не попадает в поиск;
- `restricted_in_menu` — ограниченная категория помечена для меню.

### Формат вывода

Вывод человекочитаемый. Для каждого узла нужно показать:
- `name` и/или `id`;
- положение в дереве или иерархию;
- список причин, по которым он попал в аудит.

В выводе должны присутствовать только проблемные узлы.

## Сценарий 3: SearchIndexExportVisitor

### Назначение

Подготовить JSON-экспорт для поискового индекса.

### Правила выборки

Категория попадает в экспорт, если:
- имеет тег `searchable`;
- не имеет тега `hidden`;
- не имеет тега `restricted:staff-only`.

### Формат документа

Для каждой категории формируется объект:
- `id`: string;
- `name`: string;
- `adult`: bool;
- `regions`: string[].

Правила заполнения:
- `adult = true`, если есть `restricted:18plus`, иначе `false`;
- `regions` содержит все значения из тегов `region:*`;
- если региональных тегов нет, `regions = []`.

### Формат вывода

Команда печатает JSON-массив документов в файл `export/<timestamp>-index.json`.

Требования:
- валидный JSON;
- сохранение порядка обхода;
- желательно использовать `JSON_PRETTY_PRINT` и `JSON_UNESCAPED_UNICODE`.

Пример:

```json
[
  {
    "id": "smartphones",
    "name": "Смартфоны",
    "adult": false,
    "regions": []
  },
  {
    "id": "wine",
    "name": "Вино",
    "adult": false,
    "regions": ["ru"]
  }
]
```

## CLI-контракт

Приложение должно запускаться через `php app`.

Примеры ожидаемых команд:
- `php app categories:menu <path> --adult=false --region=kg --staff=false`
- `php app categories:audit <path>`
- `php app categories:search-export <path>`

Требования:
- понятные сообщения об ошибках;
- ненулевой код завершения при ошибке;
- единообразная обработка ошибок во всех командах.

## Нефункциональные требования

- сохранить порядок узлов из JSON;
- не мутировать дерево в Visitor-ах;
- не дублировать правила без необходимости;
- корректно работать с пустыми `children`;
- доменная модель не должна зависеть от Symfony Console или JSON-формата хранения.

## Явные допущения

В спецификации зафиксированы следующие допущения:
- `root` — технический контейнер дерева, а не пункт меню;
- `regions` в поисковом экспорте содержит фактические регионы тега, а не полный список всех поддерживаемых регионов;

## Критерии готовности

Решение считается готовым, если:
- категории загружаются из JSON в объектную модель;
- дерево обходится через Visitor;
- реализованы `MenuBuilderVisitor`, `VisibilityAuditVisitor`, `SearchIndexExportVisitor`;
- каждый сценарий доступен отдельной CLI-командой;
- входные параметры валидируются;
- результаты выводятся в требуемом формате;
- код совместим с PHP 7.4;
- структура решения соответствует слоям `Domain`, `Application`, `Infrastructure`, `Bridge`.
